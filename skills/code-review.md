# 代码走读 Skill — MemoryTraceTool

> **触发**:用户说「执行代码走读 skill」「代码走读」「执行 N 轮代码走读」时,按本规范执行。
> **目标**:通过多轮全量阅读 + 编译验证 + 测试验证,系统性发现并修复工程中的问题,不允许"测试通过但功能损坏"。
> **铁律**:任一环节失败 → 修复 → 重跑,**不允许失败就退出**;每轮必须编译通过 + 测试通过 + 模拟环境通过才能进下一轮。

## 总体流程

- **默认轮次**:**10 轮**(用户可在触发时显式指定其他轮次,如「执行 5 轮代码走读」)
- **维度自动分配**:每轮开始时根据**代码改动信号 + 历史发现**自动决定本轮重点维度(详见下文「维度自动分配算法」)
- **5 个候选维度**:
  1. **并发安全**(锁粒度、原子操作、内存序、死锁、race condition、ABA、TOCTOU)
  2. **正确性**(逻辑错误、边界条件、内存安全、空指针、整数溢出、未定义行为)
  3. **跨平台兼容**(ARM32/ARM64、编译选项、ABI 差异、glibc 版本、Thumb 模式)
  4. **性能**(热路径、缓存行对齐、不必要的内存操作、锁竞争)
  5. **可维护性**(命名、注释充分性、结构清晰度、错误处理、API 边界)

## 维度自动分配算法

每轮开始时,按以下信号为 5 个候选维度评分,**得分最高的作为本轮重点维度**(同分时按上节顺序取靠前者)。

### 信号 1:本轮改动信号(权重 50%)

扫描 `git diff HEAD~1 --stat` 与本轮新引入的代码,按关键词匹配给维度加分:

| 维度 | 加分关键词 |
|------|------------|
| 并发安全 | `pthread_mutex_*`、`atomic_*`、`memory_order_*`、`compare_exchange`、`__sync_`、`volatile`、锁、CAS |
| 正确性 | `malloc`、`free`、`memcpy`、`memset`、`strcpy`、`sprintf`、`realloc`、`size_t` × 算术、NULL 解引用 |
| 跨平台兼容 | `__arm__`、`__aarch64__`、`__GLIBC__`、`__BYTE_ORDER__`、`Thumb`、`soft-float`、`gnueabi`、ABI |
| 性能 | 锁内 `printf`/I/O、`memset` 大对象、热路径、缓存行、`atomic_load` 频次 |
| 可维护性 | 函数 > 200 行、参数 > 5、嵌套 > 4、新增公开 API |

每个关键词命中 +1,该维度总分 = 命中数。

### 信号 2:历史回归风险(权重 30%)

如果前 N-1 轮的修复涉及维度 X 的代码,本轮维度 X 的回归风险升高:
- 前 1 轮涉及 → +3
- 前 2 轮涉及 → +2
- 前 3 轮涉及 → +1

例:第 1 轮修了并发问题(CAS 内存序),第 2 轮开始时「并发安全」+3 分。

### 信号 3:基础权重(权重 20%)

5 维度的固定基础权重(反映默认风险排序):

| 维度 | 基础分 |
|------|--------|
| 并发安全 | 5 |
| 正确性 | 4 |
| 性能 | 3 |
| 跨平台兼容 | 2 |
| 可维护性 | 1 |

### 信号 4:维度轮换(去重)

为了避免同一维度连续多轮(陷入单一视角),加入轮换惩罚:
- 上一轮选过的维度:-2
- 上 2 轮都选过的维度:-5(强制轮换)

### 分配示例

假设本轮改动新增 `pthread_mutex_lock` × 3、`atomic_*` × 5,前 1 轮修过并发问题:

| 维度 | 改动信号 | 历史回归 | 基础 | 轮换 | 合计 |
|------|----------|----------|------|------|------|
| 并发安全 | (3+5)×50%=4 | 3×30%=0.9 | 5×20%=1.0 | 0 | **5.9** ← 选中 |
| 正确性 | 0 | 0 | 4×20%=0.8 | 0 | 0.8 |
| 跨平台兼容 | 0 | 0 | 2×20%=0.4 | 0 | 0.4 |
| 性能 | 0 | 0 | 3×20%=0.6 | 0 | 0.6 |
| 可维护性 | 0 | 0 | 1×20%=0.2 | 0 | 0.2 |

### 10 轮维度分配建议(参考,实际按算法动态决策)

10 轮分配大致符合"高权重维度多次覆盖"的分布,典型结果:

| 轮次 | 典型维度 | 说明 |
|------|----------|------|
| 1 | 并发安全 | 改动信号最强 + 基础权重最高 |
| 2 | 并发安全 | 回归验证(若第 1 轮修复涉及并发) |
| 3 | 正确性 | 改动信号次强 + 基础权重次高 |
| 4 | 性能 | 基础权重 + 内存/锁改动信号 |
| 5 | 跨平台兼容 | ABI/类型相关改动 |
| 6 | 并发安全 | 长尾回归 |
| 7 | 正确性 | 长尾挖掘 |
| 8 | 性能 | 深度优化空间 |
| 9 | 跨平台兼容 | ARM32 实测验证 |
| 10 | 可维护性 | 文档/命名收尾 |

**算法输出**应记录到每轮 commit message 中,例如:「第 3/10 轮 — 正确性(自动分配: 改动信号 +3, 历史回归 +0, 基础 +4)」。

## 每轮执行步骤

### 步骤 1:计算本轮维度

按上述「维度自动分配算法」计算 5 维度得分,选最高者作为本轮重点。在回复中说明得分依据,让用户能看到为什么选这个维度。

### 步骤 2:全工程阅读

- 用 Read 工具遍历以下文件,**每个文件都必须完整读一遍**(不允许跳读):
  - `src/*.c` 全部源文件
  - `src/*.h` 全部头文件
  - `include/*.h` 公开头文件(若有)
  - `Makefile`(检查构建规则)
  - `examples/**/*.c`(检查示例代码是否暴露新场景)
  - `scripts/*.sh`(检查脚本逻辑)
  - `Dockerfile*`(检查构建环境)
- 用 Grep 工具针对性扫描本轮维度的常见问题模式:
  - **并发安全**:`pthread_mutex_lock`、`atomic_`、`memory_order_`、`compare_exchange`、`__sync_`、`volatile`
  - **正确性**:`malloc`、`free`、`memcpy`、`memset`、`strcpy`、`sprintf`、`realloc`、`size_t`、`int` 强转
  - **跨平台兼容**:`__arm__`、`__aarch64__`、`__GLIBC__`、`__BYTE_ORDER__`、`Thumb`、`soft-float`、`gnueabi`
  - **性能**:锁内 `printf`、锁内 I/O、长循环、`memset` 大对象、不必要的 `volatile` 读
  - **可维护性**:函数长度 > 200 行、参数 > 5 个、嵌套深度 > 4、未注释的复杂算法

### 步骤 3:列出本轮发现的问题

- 按"严重程度"排序:致命崩溃 > 内存安全 > 并发问题 > 性能问题 > 可维护性
- 每个问题给出:文件:行号、问题描述、修复方案、风险评估
- **不修复非问题** — 不做无意义的重构(参见 CLAUDE.md "Don't add features, refactor, or introduce abstractions beyond what the task requires")

### 步骤 4:逐个修复

- 改完后**立刻重读**改动的文件,确认修改正确
- 不引入新告警(`-Wall -Wextra` 必须全过)

### 步骤 5:两平台编译验证

```bash
./scripts/compile-arm64.sh clean    # 必须无告警
./scripts/compile-arm32.sh clean    # 必须无告警
```

任一失败 → 修复 → 回到步骤 4。

### 步骤 6:两平台单元测试

```bash
./scripts/compile-arm64.sh clean test    # 必须 36/36 全过
./scripts/compile-arm32.sh clean test    # 必须 36/36 全过
```

任一失败 → 修复 → 回到步骤 4。

### 步骤 7:HDM3 模拟环境测试

```bash
./scripts/sim-test.sh    # ARM32 + ARM64 综合测试必须全过
```

验收标准(任一不满足回到步骤 4):
- ARM32:`exit=0` + `entry>0` + `sites>0` + reporter 线程启动
- ARM64:`exit=0` + `entry>0` + `sites>0` + reporter 线程启动

### 步骤 8:提交本轮

```bash
git add -A
git commit -m "代码走读 第 N/{总轮次} 轮 — {维度名}(自动分配: {得分简述})

- 修复问题 1: 文件:行号 — 描述
- 修复问题 2: 文件:行号 — 描述
...

本轮无发现问题。"  # 若本轮无修复
```

**注意**:不 push,等所有轮次全过后再统一 push。

## 全部轮次完成后

### 步骤 A:统一 push

```bash
git push origin <当前分支>
```

### 步骤 B:生成代码走读报告

**必须输出**一份 Markdown 报告到 `docs/code-review-report-YYYY-MM-DD.md`(日期用当天)。

报告结构(参考 `docs/code-review-report-2026-06-14.md`):

1. **走读概览**
   - 走读日期、范围、轮次、维度顺序
   - 累计统计:总发现 / 总修复 / 遗留 / 回归 / 编译测试通过率

2. **本轮代码改动概览**
   - 走读前一次性引入的改动列表(文件 → 改动描述)

3. **问题详情**(每个问题一个小节)
   - 位置(文件:行号)
   - 严重度(致命/高/中/低)
   - 发现维度 + 发现轮次
   - 问题描述(含潜在时序/边界/数据流分析)
   - 修复方案(代码 diff 或前后对比)
   - 回归风险评估

4. **关键设计决策回顾**
   - 模式选择(如 ACTIVE/FALLBACK)
   - 锁顺序与并发不变量
   - 跨平台兼容性决策
   - 关键算法选择

5. **每轮详细验证记录**
   - 表格:轮次 / ARM64 单元测试 / ARM32 单元测试 / 模拟环境 ARM32 / 模拟环境 ARM64
   - 专项实测(API 字段、stderr 日志、跨架构 sizeof 等)

6. **维度深度审查清单**
   - 5 个维度的检查项逐一打勾(结论栏)

7. **提交历史**
   - 所有走读轮次的 commit hash + message

8. **结论与建议**
   - 走读结论(代码质量评估)
   - 后续建议(实测验证 / 后续工作 / 持续走读频率)

### 步骤 C:提交报告

```bash
git add docs/code-review-report-YYYY-MM-DD.md
git commit -m "docs: {N} 轮代码走读报告"
git push origin <当前分支>
```

### 步骤 D:向用户汇报

- 每轮发现的问题数量
- 累计修复数量
- 是否有遗留(无法在本轮解决但已记录)
- 报告文件路径
- 推送结果

## 维度检查清单(参考,非穷举)

### 并发安全维度
- [ ] 所有共享变量是否在锁/原子保护下访问?
- [ ] 锁的获取顺序是否一致(防死锁)?
- [ ] 双重检查锁定(DCL)是否用了正确的内存序(acquire/release)?
- [ ] CAS 循环是否处理了 ABA 问题?
- [ ] 锁内是否有 I/O、sleep、长循环(应该尽量挪到锁外)?
- [ ] 原子计数器是否用了正确的内存序(relaxed 还是 acquire/release)?
- [ ] 是否有 TOCTOU(检查后到使用前状态已变)?
- [ ] per_thread 槽位是否每线程独立(无跨线程写)?

### 正确性维度
- [ ] 所有 malloc/calloc/realloc 返回值是否检查?
- [ ] 所有 size_t 计算是否有整数溢出风险?
- [ ] memcpy/memset 的长度参数是否正确(源 vs 目标)?
- [ ] 字符串操作是否有缓冲区溢出风险(strcpy/sprintf)?
- [ ] 空指针是否在解引用前检查?
- [ ] 错误路径是否完整(资源是否正确释放)?
- [ ] 是否有未定义行为(违反 strict aliasing、对齐错误等)?
- [ ] 计数器 / 引用计数是否平衡(++/-- 配对)?

### 跨平台兼容维度
- [ ] 32/64 位下数据类型大小是否一致(uint64_t vs unsigned long)?
- [ ] ARM32 Thumb 模式下 bit 0 标志位是否清除?
- [ ] 字节序依赖是否消除(网络字节序 vs 主机字节序)?
- [ ] glibc 版本相关 API 是否有 fallback(malloc hooks、execinfo.h)?
- [ ] 浮点 ABI(soft/hard)是否影响数据结构布局?
- [ ] 帧指针省略(`-fomit-frame-pointer`)下 backtrace 是否仍能工作?
- [ ] 大块内存申请在 ARM32 size_t 32 位下是否溢出?

### 性能维度
- [ ] 热路径(每次 malloc/free)是否避开了不必要的操作?
- [ ] 锁粒度是否足够细(stripe locks)?
- [ ] 缓存行是否对齐(避免伪共享)?
- [ ] 内存分配是否在初始化阶段完成(避免运行时分配)?
- [ ] atomic 操作的内存序是否最宽松(relaxed > acquire/release > seq_cst)?
- [ ] 锁内是否有冗余 memset / memcpy(可挪到锁外)?

### 可维护性维度
- [ ] 每个函数是否有 doxygen 函数头?
- [ ] 关键算法/不变量是否有注释解释 WHY(不是 WHAT)?
- [ ] 函数职责是否单一(超过 200 行考虑拆分)?
- [ ] 错误信息是否对调试有帮助?
- [ ] 命名是否清晰且一致(snake_case)?
- [ ] README / CLAUDE.md 是否同步更新环境变量 / API?
- [ ] 是否有 TODO / FIXME 遗留?
